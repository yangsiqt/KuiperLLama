大模型Llama是一种先进的自然语言处理模型，它能够生成高质量的自然语言文本，用于各种应用，如对话生成、文本摘要、问答等。

LLama是一个Facebook提出的开源模型，它是一个基于 Transformer 架构的语言模型，可以用于多种自然语言处理（NLP）任务，包括但不限于，生成文章、故事、诗歌等文本内容。也可以用来做对话系统，比如回答用户提出的问题，尤其是在给定上下文的情况下，也可以用于聊天机器人和虚拟助手，与用户进行自然对话。

### 想做“不烂大街”的亮点改动：优先做这 6 类（含简历表述点）

下面这些是推理岗最认可、且能明显拉开与“跟课项目”的差距的方向。建议你选 **“1 个性能/系统亮点 + 1 个工程化亮点”** 组合。

#### 1) **连续批处理（Continuous Batching）+ KV Cache 管理**
- **做什么**：把单用户生成改成多请求队列调度；实现 KV Cache 的分配/复用/回收（必要时做 paging/块管理）。  
- **为什么值钱**：这是线上推理服务的核心能力。  
- **简历写法**：实现连续批处理调度与 KV Cache 复用，使并发场景吞吐提升 X%，显存碎片下降 Y%。

#### 2) **Paged Attention / KV Cache Paging（更进阶）**
- **做什么**：KV 按块分页存储，减少长上下文导致的显存爆炸与碎片。  
- **简历写法**：实现分页 KV 管理与按需加载，使长上下文（例如 8k/16k）下显存占用下降 X%，OOM 率下降 Y%。

#### 3) **算子融合 / FlashAttention 风格优化（MHA 重点）**
- **做什么**：把 `QK^T→Softmax→V` 融合或改成更 cache-friendly 的 kernel；或至少做 **RMSNorm+Residual**、**RoPE 融合** 等小融合。  
- **简历写法**：通过融合 kernel/减少全局内存读写，将 decode 阶段 token latency 降低 X%。

#### 4) **推理图优化：Prefill/Decode 分离 + CUDA Graph**
- **做什么**：把 prefill 与 decode 的执行路径区分；对 decode 小 batch 用 CUDA Graph 降低 launch 开销。  
- **简历写法**：引入 CUDA Graph 捕获 decode 热路径，kernel launch 开销下降 X%，p95 latency 降低 Y%。

#### 5) **更强的量化：Int4（或 FP8）+ 量化 GEMM 路径**
- **做什么**：在现有 Int8 基础上做 Int4 weight-only（常见于推理），配套解码 GEMM；或接 cublasLt/int4 gemm。  
- **简历写法**：实现 Int4 weight-only 量化推理，显存下降 X%，吞吐提升 Y%，并保证生成一致性/困惑度变化在阈值内。

#### 6) **“像线上一样”的服务化：OpenAI Compatible API + 流式输出 + 观测**
- **做什么**：做一个 HTTP/gRPC 服务，支持 SSE 流式 token；加指标（QPS、tok/s、KV 使用率、显存峰值）与 tracing。  
- **简历写法**：提供 OpenAI 兼容接口与流式输出，补齐从 engine 到 service 的闭环，并加入性能指标采集用于回归。

### 两个方向的区别（工作内容 vs 技术要求）

#### 引擎/算子（Inference Engine / Kernel）
- **日常工作**：模型加载/执行图、张量与内存管理、KV Cache、算子实现与融合、CPU/GPU 后端抽象、量化/混精、性能瓶颈定位（nsys/nvprof/perf）。
- **核心能力**：C++（模板/RAII/并发/性能）、CUDA（内存层级/warp/占用率/launch 开销）、数值稳定性（softmax/归一化）、线性代数与并行优化。
- **面试更看重**：你能不能解释清楚**token/s 为啥慢**、哪里是瓶颈、怎么做 profiling、怎么减少显存/带宽/launch 次数、以及改动带来的**可复现实测数据**。

#### 服务/系统（Inference Serving / System）
- **日常工作**：API/网关（OpenAI兼容）、流式输出、请求调度与（连续）批处理、限流与隔离、缓存/多租户、观测（metrics/log/trace）、部署与稳定性（超时/熔断/灰度）。
- **核心能力**：网络与并发（异步/线程池）、队列与调度策略、工程化与可观测性、容器化与运维（可选）。
- **面试更看重**：吞吐/时延（p50/p95/p99）、稳定性、容量评估、压测方法、线上故障处理、以及“为什么这么设计调度和缓存”。

> 你这个项目本身更偏“引擎/算子”，所以**一周、简单改造**建议走“引擎侧的可量化小增强”，最划算。
